# 超參數優化系統（Optuna）— 實施方案

## Context

目前 PPO 交易模型的超參數都是手動調整的（config.yaml + config_local.yaml），缺乏系統性搜索。模型有 ~20 個可調參數（PPO 核心、獎勵函數、交易風控），手動調參效率低且容易錯過最佳組合。需要一個自動化超參數優化流程，利用 Optuna 框架在單機（macOS Apple Silicon）上高效搜索。

## 方案

新增 `optimize.py`，使用 Optuna 框架，複用現有 `train.py` 和 `backtest/run_backtest.py` 的函數。

### 新增檔案

- `optimize.py` — 主優化腳本（~400 行）

### 修改檔案

- `requirements.txt` — 加入 `optuna`

## 設計決策

### 目標函數：加權複合分數

不用單一 Sharpe，改用多指標複合分數（符合「機構交易員」評估方式）：

```
score = 0.40 × norm(sharpe, -2, 5)
       + 0.25 × norm(return%, -20, 50)
       + 0.20 × norm(-|MDD|, -30, 0)
       + 0.15 × norm(profit_factor, 0, 3)
```

硬約束：`total_trades < 10` → 直接回傳 -1.0（模型沒學會交易）

### 分階段搜索

一次調全部 ~20 參數不現實，分三階段：

| 階段 | 參數 | Trials | 預估時間 |
|------|------|--------|----------|
| **Phase 1: PPO 核心** | learning_rate, ent_coef, gamma, n_epochs, batch_size, vf_coef | 40-50 | 5-8 hr |
| **Phase 2: 獎勵函數** | pnl_reward_scale, floating_reward_scale, take_profit_multiplier, stop_loss_extra_penalty, holding_bonus_max | 30-40 | 4-6 hr |
| **Phase 3: 精煉** | Phase 1+2 中最敏感的 3-4 個參數（縮小範圍） | 20-30 | 3-4 hr |

### 訓練步數

優化時用 **500k steps**（正式訓練的一半），比較相對表現即可，每 trial ~8-12 分鐘。

### 評估方式

用 **單次 train+backtest**（快），不用 WFA（慢 6-12x）。最終取 top 3-5 參數組合再手動跑 WFA 驗證。

## optimize.py 結構

### 1. 搜索空間定義

```python
SEARCH_SPACES = {
    "phase1_ppo": {
        "ppo.learning_rate": ("log_float", 3e-5, 5e-4),
        "ppo.ent_coef": ("float", 0.1, 0.3),        # 硬下限 0.1
        "ppo.gamma": ("float", 0.9, 0.99),
        "ppo.n_epochs": ("int", 4, 16),
        "ppo.batch_size": ("categorical", [32, 64, 128, 256]),
        "ppo.vf_coef": ("float", 0.3, 1.0),
    },
    "phase2_reward": {
        "reward.pnl_reward_scale": ("int", 300, 1000, 50),
        "reward.floating_reward_scale": ("int", 10, 80, 5),
        "reward.take_profit_multiplier": ("float", 1.0, 2.0),
        "reward.stop_loss_extra_penalty": ("float", 1.0, 8.0),
        "reward.holding_bonus_max": ("float", 0.5, 3.0),
    },
    "phase3_combined": {
        # Phase 1+2 最敏感參數，範圍從結果縮窄（執行時動態決定）
    },
}
```

### 2. 目標函數流程

```
objective(trial):
  1. 複製 base config
  2. apply_trial_params(config, trial, phase) — 用 dot-notation 設定巢狀 config
  3. 降低 total_timesteps → 500k, n_cpu → 2-4
  4. 關閉 HTML 報告、詳細 logging（加速）
  5. 複用 create_training_env(df, config) from train.py
  6. 複用 create_ppo_model(env, config) from train.py
  7. model.learn() — 搭配 OptunaPruningCallback
  8. 複用 run_backtest_pipeline(config, run_dir) 取得回測結果
  9. 計算 composite score → return
  10. 清理 env、model、臨時檔案
```

### 3. Pruning 策略（提前終止差勁的 trial）

```python
MedianPruner(
    n_startup_trials=5,      # 前 5 trial 不剪枝（建立基線）
    n_warmup_steps=100000,   # 前 100k steps 不剪枝（RL 需要暖機）
    interval_steps=50000,    # 每 50k steps 檢查一次
)
```

搭配自訂 SB3 Callback：每 50k steps 回報 mean episode reward 給 Optuna，表現低於已完成 trial 中位數即提前終止，節省 ~30-40% 運算時間。

### 4. 結果輸出

```
models/optuna/
  study_{symbol}_{timestamp}.db     # SQLite 儲存（可中斷後 resume）
  best_params.yaml                  # 最佳參數，格式同 config_local.yaml
  optimization_report.html          # Optuna 視覺化（參數重要性、搜索歷程）
```

### 5. CLI 使用方式

```bash
# Phase 1：優化 PPO 核心參數（建議 overnight 跑）
python optimize.py --phase phase1_ppo --n-trials 50

# Phase 2：固定 Phase 1 最佳參數，優化獎勵函數
python optimize.py --phase phase2_reward --n-trials 40

# 中斷後繼續（從 SQLite 恢復進度）
python optimize.py --phase phase1_ppo --n-trials 50 --resume

# 自訂訓練步數（更快但精度較低）
python optimize.py --phase phase1_ppo --n-trials 30 --timesteps 300000
```

## 關鍵注意事項

1. **ent_coef ≥ 0.1** — 低於此值模型會崩潰成「不交易」，Optuna 搜索範圍下限設 0.1
2. **資料只載入一次** — `ensure_data_ready` 在優化 loop 外執行，透過 closure 傳入 objective
3. **記憶體管理** — 每 trial 結束顯式 `env.close()` + `del model` + `gc.collect()`
4. **Multiprocessing** — `set_start_method("spawn")` in `__main__` block
5. **Verbose 靜音** — 訓練時 `verbose=0`，只印 trial 結果摘要
6. **磁碟空間** — 只保存 top N 模型檔案，其餘自動刪除

## 建議工作流程

```
Day 1 晚上：Phase 1 (PPO 核心) → 50 trials overnight
Day 2 白天：檢視結果 → Phase 2 (獎勵函數) → 40 trials
Day 2 晚上：Phase 3 (精煉) → 20-30 trials
Day 3：     取 top 3 參數組合 → 跑 WFA 驗證 → 選出最終參數

總計運算時間：~15-20 小時
日曆時間：3-4 天
```

```bash

# 2. 跑少量 trials 測試功能正常

```
